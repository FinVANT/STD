
# 🚀 AI 멀티 에이전트 플랫폼 로드맵 및 단계별 개발 순서

본 플랫폼은 초기 검증 속도를 극대화하는 MVP 단계부터 엔터프라이즈급 프로덕션 서비스로의 진화까지를 고려하여 총 3단계의 철저한 릴리즈 로드맵을 기반으로 개발을 진행합니다.

---

## 1단계. 프로토타입 (Prototype): Notion + n8n 코어 엔진 완성
> **목표**: UI 개발 공수를 최소화하고, AI 에이전트 분산 처리와 CI/CD 파이프라인의 핵심 백엔드 로직(Core Logic)을 100% 검증

가장 먼저 프론트엔드 UI 개발에 리소스를 빼앗기지 않고, 강력한 문서화 및 CMS 기능을 제공하는 **Notion을 임시 요구사항 입력창이자 관리 대시보드로 삼아 백엔드 파이프라인을 구축**합니다.

### 📋 상세 개발 순서
1. **[인프라] MySQL 큐 부트스트랩 (`v4_00`)**
   * 분산 처리를 위한 핵심 완충 레이어 테이블(`task_queue`, `component_queue`, `orchestration_event_log`)을 생성하고 인덱스를 설계합니다.
2. **[인테이크] Notion 폴링 및 자동 라우팅 파이프라인 (`v4_01`) 구축**
   * n8n `Schedule Trigger` 노드를 통해 노션 'AI Ticket 발행 DB'를 1분 주기로 모니터링합니다.
   * 요구사항 블록을 마크다운 문맥으로 덤프하고, 정규식 가중치 기반 분류 알고리즘을 실행하여 복수 개의 컴포넌트(FE/BE/DB/AI 등)를 추출한 뒤 MySQL 자식 큐에 분할 적재하는 로직을 완성합니다.
3. **[에이전트] 1:1 격리 파이프라인 및 Claude Code 연동 (`v4_03` ~ `v4_04`)**
   * 각 전문 에이전트가 MySQL 큐에서 자신의 컴포넌트에 해당하는 `READY` 상태 티켓을 배치(Bulk)로 읽어가는 워크플로우를 생성합니다.
   * 독립된 워크스페이스 내에서 Claude Code CLI를 실행하여 상용 수준의 프로덕트 코드가 누락 없이 생성되도록 시스템 프롬프트를 미세 조정합니다.
4. **[배포 및 콜백] GitHub PR 발행 및 Actions CI/CD 연동 (`v4_05` ~ `v4_07`)**
   * 생성된 코드를 독립 브랜치로 푸시하고 GitHub API를 통해 Pull Request를 자동 발행하는 노드를 연결합니다.
   * GitHub Actions의 빌드/테스트 완료 Webhook 콜백을 수신하여, MySQL 큐와 Notion의 최종 상태를 `완료` 또는 `실패`로 원자적(Atomic)으로 갱신하는 통합 업데이트 엔진을 완성합니다.

## 2단계. 베타 (Beta): 자체 웹사이트 완성 & 아키텍처 기반 혁신 증명
> **목표**: 사용자 전용 Web Client를 도입하여 일반 비개발자 접근성을 확보하고, 기존 아키텍처의 유연성과 확장성을 해커톤 무대에서 완벽히 증명

백엔드 코어 엔진이 완벽히 검증되었으므로, 일반 사용자가 접속하여 요구사항과 STT 텍스트를 입력할 수 있는 **전용 웹사이트(Web Client)**를 개발합니다. 이때 기존 노션 보드는 **"관리자용 백오피스 관제 페이지(Back-Office Dashboard)"**로 자연스럽게 역할이 격리 승격됩니다.

### 📋 상세 개발 순서
1. **[인테이크 추가] Spring Webhook 인테이크 (`v4_02`) 구축**
   * 웹 클라이언트가 백엔드 API를 통해 실시간으로 데이터를 밀어 넣을 수 있도록 n8n 내에 `POST /nova/v4/web-intake` 실시간 Webhook 수신 레이어를 추가합니다.
2. **[선행 적재 연동] Web-to-Notion 자율 생성 흐름 구현**
   * 웹사이트로 요청이 인테이크되는 즉시 n8n이 고유 `Ticket ID`를 생성하고, **Notion API를 원격 호출하여 관리자 DB에 새 페이지를 선행 생성**하도록 구현합니다.
   * 이로써 웹 유입 건과 노션 직접 등록 건이 단일화된 관리자 대시보드(Notion CMS)로 통합됩니다.
3. **[프론트엔드] 전용 Web Client UI/UX 개발**
   * 사용자가 아이디어나 프로젝트 요구사항 명세를 텍스트로 기입하는 가볍고 직관적인 웹 화면을 빌드합니다.
   * 제출 버튼 클릭 시 위에서 개방한 n8n Webhook 엔드포인트로 JSON 페이로드를 전송하도록 프론트엔드 연동을 마칩니다.
4. **[통합 테스트] End-to-End 시나리오 및 시연 환경 검증**
   * `Web Client 입력` ➔ `n8n Webhook 수신` ➔ `Notion 백오피스 자동 기재` ➔ `MySQL 완충 큐 적재` ➔ `멀티 에이전트 자율 코딩` ➔ `GitHub Actions 배포` ➔ `Notion 결과 최종 피드백`으로 이어지는 전체 흐름을 최종 튜닝합니다.

## 3단계. 고도화 (Production): 외부 의존성 제거 및 Notion 완벽 제거
> **목표**: 외부 CMS API(Notion)의 제한 사항을 100% 탈피하고, 대규모 트래픽과 기업 보안 요구사항을 만족하는 엔터프라이즈급 완전 자립형 프로덕션 서비스로의 진화

해커톤 이후의 실제 상용화 및 대규모 서비스 가동을 위한 고도화 단계입니다. 노션 API의 태생적 한계(초당 요청 제한, 외부 클라우드 보안 리스크)를 제거하기 위해 노션을 아키텍처에서 완전히 걷어냅니다.

### 📋 상세 개발 순서
1. **[백엔드 고도화] Spring Boot 핵심 API 서버 레이어 내재화**
   * n8n Webhook이 직접 웹 트래픽을 받는 구조에서, 자체 백엔드 애플리케이션(Spring Boot)이 모든 웹 요청 및 사용자 인증 메커니즘을 전담하는 안정적인 구조로 고도화합니다.
2. **[디비 직결] 자체 MySQL 완충 큐 Direct Insert 전환**
   * 웹사이트에서 사용자가 요구사항을 제출하면, Spring Boot 백엔드가 서비스 내부의 **MySQL DB `task_queue` 테이블에 데이터를 0.001초 만에 직접 삽입(Insert)** 하도록 수정합니다. 외부 노션 API 호출로 인한 네트워크 지연과 레이트 리밋 병목이 원천 소멸됩니다.
3. **[n8n 트리거 변경] DB 기반 비동기 폴링 전환**
   * n8n 오케스트레이터의 최초 진입점(Trigger)을 노션 감시에서 **`MySQL task_queue 내 READY 레코드 감시`**로 전환합니다.
   * n8n은 오직 순수한 워크플로우 오케스트레이션 및 상태 머신(State Machine) 역할에만 집중하게 되어 엔진 전체가 극도로 가벼워집니다.
4. **[어드민 내재화] 자체 백오피스 어드민 대시보드 UI 개발**
   * 기존에 노션 칸반 보드로 수행하던 모든 관제 영역(태스크 진행 스태터스, 에러 로그 문맥 확인, GitHub PR 링크 포워딩 등)을 웹사이트 내부의 **'관리자 전용 대시보드' 웹 UI로 완전 내재화**하여 시스템 자립도를 100% 달성합니다.
